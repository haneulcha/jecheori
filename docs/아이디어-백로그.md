## 정적·무추적을 유지한 채 가능한 것들

### 2. 등급별 가격 🟡
- **왜:** 지금은 '상품'(최상 등급)만 보여준다 — 실제 장바구니보다 가격이 높게 잡힌다.
- **가능성:** KAMIS 응답에 이미 `rank`가 온다. 우리가 '상품'만 고르고 버릴 뿐. 스냅샷에 전
  등급을 남기고 카드에 등급 토글/비교를 붙이면 된다.
- **먼저 확인:** 소매(`p_product_cls_code='01'`)에 등급이 실제로 몇 개 오는지. 상품/중품 둘뿐이면
  토글보다 "중품 대비 −N%" 한 줄이 나을 수 있다.
- **얽힘:** `schemaVersion` 가드(아래) 먼저.

### 3-수산. 수산물 확대 🟡 — 추천
- **왜:** 계절성이 이 앱의 심장인데, 수산물은 계절성이 가장 뚜렷하다(봄 주꾸미, 가을 전어,
  겨울 굴·방어). 제철 달력에 이보다 잘 맞는 카테고리가 없다.
- **가능성:** `fetch-prices.mjs`의 `CATEGORY_CODES`에 `600`(수산물) 추가 + 프로필 작성.
- **함정:** 새 단위(마리·근)를 몰고 온다. `parseUnit`은 처음 보는 단위에 **일부러 throw**한다
  (조용한 오염 방지). 단위를 가르치기 전까진 수집이 시끄럽게 실패한다 — 안전장치이므로 정상.

### 4. 조사일 상시 표시 🟢 — 값싼 정직함 ✅ **완료 (2026-07-16)**
- **원 아이디어:** "품목별로 며칠 기준인지 보여주기."
- **반전:** 품목별 날짜는 필요 없다. 스냅샷은 **조사일 하나**를 갖고 그날 값 없는 품목은
  `null`로 빼므로(다른 날로 안 메움), **화면의 모든 가격은 같은 하루의 값**이다.
- **한 것:** 그 하루를 **항상** 노출한다 — 헤더 제목 아래 `오늘 · 7월 13일 기준`(상대+절대).
  3일 임계(`STALE_AFTER_DAYS`)와 stale 경고를 없애고 `Freshness`를 `none|dated`로 재설계,
  표시는 `src/week.ts` `surveyedLabel`. 스냅샷 없으면 줄 없음. 스펙·플랜: `2026-07-16-survey-date-*`.
- **품목별 날짜는** '결측을 다른 날 값으로 메우는' 정책으로 바꿀 때만 의미가 생긴다(안 바꿈).

### 6. 커버리지 진단 페이지 🟢
- **왜:** `npm run report:coverage`(어떤 제철 품목이 가격 매칭에 성공/실패했나, itemName 불일치냐
  kindName 불일치냐)를 브라우저에서 보고 싶다.
- **가능성:** 로직이 이미 있다. 라우트 하나(`/coverage` 등) 붙이면 된다.
- **주의:** 정적 프리렌더라 "숨김 경로"도 URL만 알면 누구나 접근한다(=링크만 안 걸린 것, 비공개
  아님). 커버리지 정보는 민감하지 않아 괜찮지만, 크롤 방지는 별도(robots/프리렌더 제외).

## 무개인화 경계를 넘는 것 (phase 2 — 서버/개인화 사이클)

### 1. 지역별 가격 + GPS 🔴
- **왜:** 유저 위치 기준 가격.
- **가능성:** KAMIS `p_country_code`에 지역 코드(서울 1101·부산 2100 등)를 넣으면 지역 데이터는
  받아진다.
- **충돌:** GPS+위치 기준 가격은 이 앱의 세 기둥(정적 프리렌더·무추적·무개인화)과 정면충돌.
  CLAUDE.md가 "개인화는 다음 사이클"로 그어둔 선이다.
- **가벼운 버전(지금도 가능):** GPS 없이 주요 지역 몇 개를 프리렌더 + 사용자가 드롭다운으로 선택
  (추적 없음, 정적 유지).
- **먼저 확인:** KAMIS 지역 소매는 전국 평균보다 품목이 훨씬 적다 → 지역 쓰면 무가격 카드가 늘 수 있다.

### 3-축산. 축산물 확대 🔴 — 개념 재설계 필요
- **문제:** 계란·우유·소고기는 계절이 없다. "이번 달 제철 소고기?"는 어색하다.
- **판단:** 넣으려면 계절 달력이 아니라 다른 틀(가격 추이만)이 필요하다. 수산물을 먼저 하고,
  축산물은 앱의 개념을 다시 잡을 때.

## 검토 결과 "그대로 둔다"로 정리된 것

### 5. 레시피 품목 간 중복 제거 — 무해함 (오해였음)
- **의심:** 두 품목이 같은 레시피를 참조하면 한쪽에서 사라지지 않나?
- **사실:** 안 사라진다. 중복 제거는 **JSON 저장을 한 번만** 하는 것이고, 매칭(`matchRecipes`)이
  레시피 **이름으로 전체 목록을 훑어서** 참조하는 모든 품목 카드에 다 뜬다. 저장 최적화지 표시
  정책이 아니다. 그대로 둔다.

## 이 아이디어들을 건드리기 전 공통 선결 과제

- **`schemaVersion` 가드:** 스냅샷은 `schemaVersion: 2`인데 로더(`src/data.ts`)가 버전을 검사하지
  않는다. 등급·지역·날짜를 데이터 모델에 추가하면(2·1·4번) 구조가 바뀌므로, **버전 가드를 먼저**
  넣어야 옛 JSON과 새 코드가 조용히 어긋나는 걸 막는다.
- **정적 산출물 크기:** 프리렌더가 스냅샷을 전부 굽는다(지금 80항목). 등급 × 지역으로 곱하면
  JSON·빌드시간이 급증한다. 1·2번을 합칠 땐 크기를 본다.
- **우선순위 제안:** 정적·무추적을 지키는 4🟢 → 6🟢 → 3-수산🟡 → 2🟡를 먼저, 경계를 넘는
  1🔴·3-축산🔴은 phase 2로 분리.
