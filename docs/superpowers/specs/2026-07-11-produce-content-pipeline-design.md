# 과채 데이터 수집·에디토리얼 콘텐츠 파이프라인 설계

> 2026-07-11 · 브레인스토밍 산출물
> 관련: `CONTEXT.md`(도메인·심), `CLAUDE.md`(규칙·배포), `DESIGN.md`(팔레트·톤),
> `docs/superpowers/specs/2026-07-10-card-info-and-price-display-design.md`(카드 설계)

## 배경

앞으로 과일·채소에 **궁합(페어링)** 과 **간단 레시피**를 붙이고 싶다. 그런데 이건
"필드 하나 추가"가 아니라, 가격이 이미 그렇듯 **콘텐츠도 파이프라인으로** 만들고 싶다는
요구다. 그래서 먼저 물었다 — *"과채 관련 데이터·콘텐츠를 어디서, 어떻게 가져오나?
Open API에만 의존해도 되나?"*

이 앱은 이미 **콘텐츠를 Open API에 의존하지 않는다.** `produce.json`의
`whyNow`/`howToPick`/`howToStore`/`howToUse`는 전부 손으로 쓴 에디토리얼이고, KAMIS
Open API는 **가격 하나만** 담당한다(`prices.json`). 궁합·레시피는 성격상 가격이 아니라
이 에디토리얼 계열에 가깝다.

리서치(deep-research, 103 에이전트 교차검증)와 실 API 실측으로 데이터원의 실체를
확인했고, 그 결과가 설계를 결정했다. 이 문서는 그 결론을 굳힌다.

## 원칙 (유지)

- **공개 페이지는 경량·무추적·런타임 외부요청 0.** 수집·가공은 CI/빌드 타임에만, 앱은
  정적 JSON만 읽는다. (KAMIS 파이프라인과 동일 모양.)
- **사용자 문구는 담백한 한국어, 이커머스 화법 금지.** 궁합·레시피도 예외 없음.
- **API 키는 코드·저장소에 절대 넣지 않는다.** 환경변수·CI 시크릿으로만
  (`KAMIS_CERT_KEY`와 동일 관례).
- **재배포 데이터는 출처표시 의무를 진다** (KOGL 1유형). 화면 어딘가에 출처를 명시한다.
- **순수 로직은 `picks/card/app`, 표시는 `components`.** 새 데이터원·콘텐츠도 이 경계
  안으로 들어온다.

## 리서치·실측 결과 (설계 근거)

### 데이터원 카탈로그

| 소스 | 무엇을 주나 | 접근 | 라이선스·재배포 | 우리 쓰임 |
|---|---|---|---|---|
| **식약처 식품영양성분DB** (data.go.kr 15127578) | 영양성분·원산지 | 무료 REST (`apis.data.go.kr/1471000/FoodNtrCpntDbInfo02`) | 제한없음(KOGL1, 출처표시) → **재배포 OK** | 궁합·레시피 글의 **근거(grounding)** |
| **식약처 조리식품 레시피DB** (COOKRCP01, data.go.kr 15060073 / 식품안전나라) | 재료·조리단계·1인분 영양·이미지 | 무료 REST (`openapi.foodsafetykorea.go.kr/api/{KEY}/COOKRCP01`) | 제한없음(KOGL1, 출처표시) → 재배포 OK | 채소 레시피의 **근거·영감** |
| **농진청/농사로 품종정보** (data.go.kr 3060745 · 15087193) | 품종 용도·특성·재배 | 무료 REST | 제한없음 → 재배포 OK | 품종 디테일 grounding (선택) |
| **음식 궁합/페어링** | — | — | — | **소스 없음** (한국·글로벌 모두) |

제외한 것: **Spoonacular**(저장·캐싱 금지 → 정적 재배포와 비호환),
**TheMealDB**(한식 0건), **FlavorDB2**(CC BY-NC-SA 비상업 → 상업 사용 불가),
**aT/KAMIS**(가격·유통만, 제철/궁합/영양 데이터셋 없음).

### 실측 1 — 영양DB에서 원물 추출 (2026-07-11, 실 키 검증)

- `FOOD_NM_KR` 부분검색만으론 원물이 안 잡힌다. "사과" 검색 → `totalCount 2832`,
  상위 100건이 죄다 음료·과자·빵(사과파이·사과와플…). **가공식품 편향 확인.**
- **`FOOD_CAT1_NM=과일류`(또는 채소류) 필터가 동작한다.** "사과 & 과일류" → 8건,
  전부 원물(`사과_생것`·`사과_부사_생것`·`사과_홍옥_생것`…). (참고: `DB_GRP_NM`
  파라미터는 무시됨.)
- **결론**: 품목별로 `FOOD_CAT1_NM` 필터 + 이름 → 원물 후보 몇 개 → **대표 엔트리 하나를
  수기 지정**하면 깨끗한 영양값을 얻는다. `KamisRef`(품목→item_name 수기 매핑)와 같은 패턴.

### 실측 2 — 레시피 커버리지 (COOKRCP01 전체 1,146건, 40품목 스윕)

품목이 **재료로 걸리는 수**(조연 포함)와 **레시피 이름에 등장하는 수**(주인공 프록시)를 측정.

- **채소는 조리 레시피가 풍부** — 무 59·토마토 52·단호박 36·고구마 33·가지 27·배추 24·
  감자 18·시금치 14(이름 등장 기준).
- **생과일은 거의 없음** — 수박 5·딸기 3·참외 4·복숭아 2·감귤 2·멜론 1·참다래 0·
  샤인머스캣 0. 예외는 배 37·사과 11(요리에 쓰임).
- **이건 데이터 공백이 아니라 현실**이다. 여름 생과일은 조리하는 게 아니라 그냥 먹는다.
  대신 궁합·서빙 아이디어가 어울린다.
- ⚠️ 이름 필터는 부분문자열이라 짧은 이름(무·배·가지)에 오탐이 섞인다. 절대수치는 느슨히,
  **"생과일≈0, 조리채소 충분"이라는 거시 패턴**만 단단한 신호로 본다.

## 핵심 설계 결정

### 결정 1 — 콘텐츠가 분류별로 갈린다

실측이 두 콘텐츠를 품목 분류에 자연스럽게 대응시켰다.

```
🍑 과일  → "한 줄 궁합" 우선   (레시피 소스 없음 → 순수 에디토리얼)
🥬 채소  → "간단 레시피 카드" 우선  (COOKRCP01로 실제 grounding 가능)
```

- 한 품목에 둘 다 억지로 채우지 않는다. **분류에 맞는 걸 우선** 배치한다.
- 겹치는 품목(배·사과처럼 요리에도 쓰이는 과일)은 둘 다 가질 수 있다 — 강제 아님.
- 앞서 우려한 "COOKRCP01 저염 급식 톤"은 채소에서 문제되지 않는다. 완제품으로 뿌리지
  않고 **에디토리얼 카드의 근거/영감**으로만 쓰기 때문. 과일은 애초에 레시피를 안 쓴다.

### 결정 2 — 모든 소스는 "수기 참조 레이어 + 그 위 에디토리얼"

세 소스를 다 두드려보니 일관된 패턴이 나왔다.

| 소스 | 자동 수집? | 실제 필요한 것 |
|---|---|---|
| 영양성분DB | 반자동 | 품목→대표 원물 엔트리 **수기 매핑** |
| 레시피DB | 반자동 | 채소별 레시피 **사람 선별**(근거·영감) |
| 궁합 | 불가 | **순수 에디토리얼** (소스 없음) |

이는 앱이 **이미 하는 방식**이다 — `KamisRef`(수기 매핑) + 손으로 쓴 프로필. 새 데이터원도
같은 심에 얹는다. **자동 파이프라인이 콘텐츠를 만들지 않는다. 자동은 근거를 모으고,
콘텐츠는 사람이 (AI 보조로) 쓴다.**

## 아키텍처

앱 아키텍처를 흔들지 않는다. `picks ← card ← app ← components ← routes` 경계와
"수집은 CI, 앱은 정적" 원칙을 그대로 확장한다.

```
[수집 계층 · CI]
  fetch:prices     → prices.json      (기존)
  fetch:nutrition  → nutrition.json   (신규 · foodDbRef 기반 배치 수집)
  fetch:recipes    → recipes.json     (신규 · 선택 · 채소 근거용)
        │
[에디토리얼 계층 · 사람 + AI 보조]
  수집 근거 + 손글  →  produce.json 콘텐츠 필드 확장
    · 과일: pairings (궁합 한 줄들)
    · 채소: recipes  (간단 레시피 카드)
        │
[조립 · 순수]   buildAppView → AppView
[투영 · 순수]   toCardView   → CardView (새 콘텐츠 투영)
[표시]          components   → JSX (표시만)
```

### 데이터 모델 변경 (`src/types.ts`)

참조 레이어에 영양 참조를 추가한다(`KamisRef` 옆).

```ts
/** 식약처 영양DB에서 원물 하나를 집기 위한 수기 참조.
 *  FOOD_CAT1_NM 필터 + 대표 엔트리로 매칭 (KamisRef와 같은 패턴). */
export interface FoodDbRef {
  /** '과일류' | '채소류' — FOOD_CAT1_NM 필터값 */
  category1: string
  /** 대표 원물 엔트리명 (예: "사과_부사_생것") 또는 FOOD_CD */
  foodName?: string
  foodCode?: string
}

export interface ProduceProfile {
  // ...기존 필드...
  kamis: KamisRef
  foodDb?: FoodDbRef              // 신규 — 영양 grounding
  /** 궁합 한 줄들 (주로 과일). 에디토리얼. */
  pairings?: string[]
  /** 간단 레시피 카드 (주로 채소). 에디토리얼. */
  recipes?: SimpleRecipe[]
}

/** 간단 레시피 카드 — 3~5 재료, 3~4 단계. 에디토리얼(수집 근거로 작성). */
export interface SimpleRecipe {
  title: string
  ingredients: string[]
  steps: string[]
  /** 근거 출처 표기용 (예: "식약처 조리식품 레시피 DB 참고") */
  source?: string
}
```

- 수집 산출물(`nutrition.json`)은 별도 스냅샷 타입으로 두되, `PriceSnapshot`과 같은 결을
  따른다(`schemaVersion`·`fetchedAt`·`entries`).
- `pairings`·`recipes`는 **에디토리얼이라 `produce.json`에 직접** 들어간다(손글 프로필과
  동거). 수집 데이터가 아니다.

### 표시 심 (`src/card.ts` → `src/components/`)

- `toCardView`가 `pairings`·`recipes`를 `CardView`에 투영한다(개당값·스파크처럼 순수 파생).
  표시 여부·정렬 같은 규칙은 여기서 끝난다.
- 컴포넌트는 `CardView`를 JSX로만 그린다. 한국어 카피·마크업·이스케이프는 컴포넌트 소관.
- 카드 밀도를 위해 기존 `<details>` 접힘 구조 안에 배치(레시피는 펼침, 궁합 한 줄은 노트
  영역). 구체 레이아웃은 구현 시 `DESIGN.md` 규율로 확정.

### 수집 스크립트 (`scripts/`)

- **두 소스는 인증 시스템이 다르다**(실측 확인). 영양DB는 **data.go.kr 게이트웨이**
  (`apis.data.go.kr`, 계정 단위 serviceKey), 레시피는 **식품안전나라**
  (`openapi.foodsafetykorea.go.kr`, 별도 키). 키 하나로 양쪽을 못 쓴다.
- `fetch-nutrition.mjs` — data.go.kr serviceKey(예: `DATA_GO_KR_KEY`)로 `foodDb` 참조가
  있는 품목만 `FOOD_CAT1_NM` 필터 배치 조회 → `public/data/nutrition.json` 커밋.
  `fetch-prices.mjs`와 동일 골격(env 검사·실패 시 명확한 에러·스냅샷 커밋).
- `fetch-recipes.mjs`(선택) — 식품안전나라 키(예: `FOODSAFETY_RECIPE_KEY`)로 채소 품목의
  후보 레시피를 수집해 **에디토리얼 작성용 근거**로 저장. 완제품이 아니라 초안 재료다.
  워크플로(CI 커밋 vs 로컬 도구)는 별도 결정.
- CI 배선은 `update-prices.yml` 패턴 그대로:
  `SOME_KEY: ${{ secrets.SOME_KEY }}`.

## 라이선스·출처표시

재배포하는 수집 데이터(영양·레시피)는 KOGL 1유형이므로 **출처표시 의무**가 있다.

- 화면에 출처를 명시한다: 예) 영양 정보 옆 "출처: 식품의약품안전처", 레시피 근거
  "식약처 조리식품 레시피 DB 참고".
- 문구·위치는 `DESIGN.md` 톤에 맞춰 담백하게. 이커머스 배지처럼 요란하지 않게.

## 범위 밖 (YAGNI)

- **완제품 레시피 API 실시간 호출** — 원칙(무외부요청)·톤과 충돌, 제외.
- **분자/향미 페어링 데이터** — FlavorDB2 비상업 라이선스로 사용 불가.
- **모든 품목에 궁합+레시피 둘 다** — 분류별 우선 배치로 충분. 강제하지 않는다.
- **품종정보·EPIS 레시피재료** — 지금은 안 붙인다. 필요해지면 같은 참조 패턴으로 추가.
- **로그인·개인화** — 다음 사이클. 이 설계는 정적 범위 안에서 완결.

## 미검증 지점 (구현 시 확인)

- **채소 원물 필터** — 과일류로 검증한 `FOOD_CAT1_NM` 필터가 채소류에서도 깨끗한지
  실측(오이·시금치·무 등). 스키마는 동일하므로 거의 확실하나 확인.
- **EPIS 레시피재료 API(15058981)** — 실제 재료·영양 필드 범위 미검증(리서치에서 1-2 반박).
  붙이기 전 스키마 직접 확인. LINK형이라 만개레시피 ToS 별도 적용 가능.
- **레시피 근거 워크플로** — `fetch:recipes`를 CI 커밋으로 둘지, 로컬 에디토리얼 도구로
  둘지 미정. 첫 슬라이스에서 결정.
- **제철 시기 1차 출처** — 별도 공개 소스가 애매하나, 앱은 이미 `seasonMonths`를 수기
  큐레이션하므로 공백 아님.

## 구현 순서 (tracer bullet)

작게 관통해 파이프라인 뼈대를 먼저 세운다.

1. **영양 grounding 슬라이스** — `FoodDbRef` 타입 + 품목 2~3개 수기 매핑 +
   `fetch-nutrition.mjs` + `nutrition.json` + 카드에 영양 한 줄 표시. 수집→매핑→표시
   전 구간을 한 바퀴. (소스가 가장 확실해 뼈대 검증에 최적.)
2. **과일 궁합 슬라이스** — `pairings` 필드 + 과일 몇 품목 에디토리얼 작성 + 카드 표시.
   수집 없는 순수 에디토리얼 경로 검증.
3. **채소 레시피 슬라이스** — `SimpleRecipe` + COOKRCP01 근거로 채소 몇 품목 카드 작성.

각 슬라이스는 독립적으로 배포 가능한 세로 조각이다. 상세는 구현 계획(writing-plans)에서.
