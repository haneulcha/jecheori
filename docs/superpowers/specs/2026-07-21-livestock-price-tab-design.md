# 축산물 값 탭 설계

> 2026-07-21 · 브레인스토밍 산출물
> 관련: `docs/superpowers/specs/2026-07-12-coming-seasonal-index-tab-design.md`(별도 탭 선례),
> `docs/superpowers/specs/2026-07-20-seafood-expansion-design.md`(카테고리 확장 선례),
> `docs/제품-동작-지도.md`(수집 정책·지렛대 지도), `CLAUDE.md`(데이터·정책 결정 규칙)

## 배경

이 앱은 "이번 달 **제철**인 것 중 값이 내려온 것"을 알려주는 계절 달력이다. 과일·채소·수산물은
자연 제철이 있어 달력에 섞인다. 그런데 **축산물(소·돼지·닭·달걀)은 대부분 연중 사육이라
자연 제철이 없다.** 삼겹살에 제철은 없다.

사용자는 축산물을 추가하되, 축을 **제철이 아니라 가격 움직임**으로 잡기로 했다. 곧 축산물은
"제철"을 주장하지 않고 "요즘 값이 내려온 것"만 이야기한다. 제철 달력의 무결성을 지키기 위해
**제철 그리드에 섞지 않고 별도 탭**(`/coming`과 같은 선례)에 둔다.

이 기능은 "거울 아닌 창"(제철 창) 철학의 경계선에 있다. 시세 티커가 되지 않도록, 화면 톤은
**장보기 참고**에 머문다 — 이커머스 화법 금지, 담백한 한국어, 제철을 사칭하지 않음.

## 결정 요약 (브레인스토밍에서 확정)

| 결정 | 값 | 근거 |
|---|---|---|
| 표시 축 | **가격만** (제철 무시) | 축산물은 자연 제철이 없음 |
| 화면 배치 | **별도 탭/라우트** | 제철 달력과 섞으면 "왜 이번 달 제철?" 오해 |
| 품목 범위 | **가급적 전체** (KAMIS 부류 500 소매) | 장보기에서 실제 사는 것 위주, 최대 커버 |
| 소고기 등급 | **등급 전부 포함**(1·2·3 등) | 등급별로 값이 갈림 — 부재로 숨기지 않음 |
| 소고기 카드 모델 | **등급마다 별도 카드** | 기존 "한 프로필=한 카드=한 가격" 재사용 |

## 원칙 (유지)

- 런타임 의존성 0, 런타임 외부요청 0. 가격은 CI 커밋 JSON, KAMIS 호출은 CI(Lambda)에서만.
- 사용자 문구는 담백한 한국어. 이커머스 화법 금지("사세요" ✕). **"제철" 표현 금지**(축산물엔 없음).
- KAMIS 매칭은 품목 코드가 아니라 `item_name`·`kind_name` **문자열**로.
- 순수 로직은 `picks/card/app`에, 표시는 `components`에.

## 데이터 모델 — 기존 구조 재사용

축산물도 **`ProduceProfile`**로 표현한다. 새 타입을 만들지 않는다.

- `Category` 유니온에 `'livestock'` 추가 (`src/types.ts`).
- `KamisRef.categoryCode` 유니온에 `'500'` 추가 (현재 `'100' | '200' | '400' | '600'`).
- 축산물 프로필의 필드 관례:
  - `category: 'livestock'`
  - `kamis: { categoryCode: '500', itemName: …, kindName: … }`
  - `seasonMonths: []`, `peakMonths: []` — **제철 없음**. 축산물 뷰는 계절로 게이팅하지 않으므로
    빈 배열로 둔다. (이 빈 배열이 제철 라우트로 새지 않도록 §"심 주의" 참고.)
  - `whyNow: { default: "<부위·등급 한 줄 설명>" }` — 월별이 아니라 **정적 한 줄**. 기존
    `whyNowLine`의 `default` 폴백을 그대로 탄다. 사실 기반(등급 특성·쓰임), 인상 클리셰 금지.
  - `howToPick / howToStore / howToUse` — 그대로 유용하니 채운다.
  - `foodDb` / `recipeRef` — 있으면 붙이고(달걀·닭고기 등 매칭되면), 없으면 생략(선택 필드).

### 소고기 = 등급마다 별도 프로필

소고기는 하나의 품목이 아니라 **등급·부위별 여러 프로필**로 쪼갠다. 예:

- `한우 등심 1++등급`, `한우 등심 1+등급`, `한우 등심 1등급`, `수입 소고기(갈비/불고기)` …

각 프로필은 `kamis.kindName`으로 KAMIS의 품종(등급) 문자열을 매칭한다. `name`/`emoji`는
사람이 읽을 라벨(예: `🐂 한우 등심 1등급`). 카드·가격 파생은 기존 기계를 그대로 쓴다.

> **실제 KAMIS 품목/품종 문자열은 추측하지 않고 응답으로 확정한다** — 수산물 때
> `scripts/probe-seafood.mjs`로 확인한 것처럼. 프로필의 `itemName`/`kindName`을 실측으로
> 채우는 것이 플랜의 **첫 단계**다. 아래 품목 목록은 KAMIS 응답 확인 전의 잠정안이다.

### 잠정 품목 목록 (실측으로 확정)

| 라벨(잠정) | KAMIS itemName(잠정) | kindName(잠정) |
|---|---|---|
| 한우 등심 1++등급 | 쇠고기 | 한우 등심(1++등급) |
| 한우 등심 1+등급 | 쇠고기 | 한우 등심(1+등급) |
| 한우 등심 1등급 | 쇠고기 | 한우 등심(1등급) |
| 수입 소고기 | 쇠고기 | 수입갈비 / 수입불고기 |
| 삼겹살(국산) | 돼지고기 | 삼겹살 |
| 목살(국산) | 돼지고기 | 목살 |
| 닭고기 | 닭고기 | (브로일러) |
| 계란(특란) | 계란 | 특란 10개 |

오리고기 등 소매 조사 여부가 불확실한 품목은 응답에 있으면 추가, 없으면 뺀다(빠뜨린 이유는
플랜/코드 주석에 남긴다 — 부재를 조용히 굳히지 않기).

## 화면·라우팅 — `/coming` 선례 그대로

- 새 라우트 **`/livestock`** (`src/routes/livestock.tsx`): 로더가 `produce.json`+`prices.json`+
  `nutrition.json`+`recipes.json`을 읽어 `buildLivestockView`로 조립, 프리렌더.
- 새 순수 함수 **`buildLivestockView`** (`src/app.ts`): `category === 'livestock'` 프로필만 골라
  KAMIS 매칭 → 카드. **제철 게이팅 없음.** 정렬은 `sortCards(cards, 'drop')`(가장 많이 내린 순).
- 새 컴포넌트 **`Livestock`** (`src/components/Livestock.tsx`): 카드 그리드 재사용. 상단 카피는
  제철이 아니라 "요즘 값이 내려온 축산물" 류. 절정(peak) 뱃지·제철 스트립 등 계절 UI는 뜨지
  않게 한다(축산물 카드엔 `inPeak=false`, 제철 월 없음).
- **`NavIndex`**: `current` 타입에 `'livestock'` 추가, 드로워에 링크(예: "축산물 값") 추가.

### 선정·정렬·결측 규칙 (데이터 정책)

- **선정**: `category==='livestock'` 프로필 전체가 후보. 제철 조건 없음.
- **정렬**: `sortCards(..., 'drop')` — 한 달 전 대비 하락폭 큰 순. 기존 정렬 재사용.
- **결측(오늘 조사 없음)**: 가격이 `null`인 항목도 **숨기지 않고** 명단 뒤로 가라앉혀 "오늘 조사
  없음"으로 표시(기존 카드의 null 처리 톤과 동일). 명단을 안정적으로 유지 — 사라졌다 나타나는
  깜빡임 방지.

## 핵심 심(seam) 주의 — 제철 라우트로의 누수 방지

`buildAppView`(지금 제철)·`buildComingView`(다가오는)는 **프로필 배열 전체**를 순회한다
(`selectPicks`, `searchIndex`, `comingMonths`). 축산물 프로필을 `produce.json`에 함께 두면
`seasonMonths: []`라도 이 이음매로 새어 검색 인덱스·다가오는 목록에 나타날 수 있다.

→ **이음매에서 카테고리로 거른다:**

- `buildAppView` / `buildComingView`: 입력 프로필을 `category !== 'livestock'`로 필터.
- `buildLivestockView`: `category === 'livestock'`만.

(수산물은 제철이라 지금처럼 달력에 섞인다 — 축산물만 예외다.)

## 수집 파이프라인

- `scripts/fetch-prices.mjs`의 `CATEGORY_CODES`에 `'500'` 추가 → `prices.json`에 축산물 포함.
  Lambda(`scripts/lambda/index.mjs`)·EventBridge는 그대로(같은 스크립트 경로).
- `scripts/fetch-coming-prices.mjs`: **손대지 않는다** — 축산물은 제철이 없어 "다가오는" 개념이
  없다(coming-prices는 작년 이맘때 씨앗용).
- `report:coverage`(`scripts/report-coverage.mjs`): 부류 500도 리포트하도록 확장 — 제철 프로필↔
  가격 매칭처럼 축산물 프로필↔가격 매칭 커버리지를 본다.
- **게이트**: 수산물 때처럼, KAMIS 500 응답을 **로컬/Lambda 수동 invoke로 200 확인**한 뒤에
  품목 문자열을 확정하고 EventBridge 변화(없음 — 코드만 확장)를 반영한다.

## 테스트

- **순수 로직**(`tests/`): `buildLivestockView`가 (a) 축산물만 고르고 (b) 제철 관계없이 전부
  후보로 넣고 (c) 하락순 정렬하며 (d) null 가격을 뒤로 두는지. `buildAppView`/`buildComingView`가
  축산물을 **배제**하는지(누수 회귀 테스트).
- **컴포넌트**(`src/components/Livestock.test.tsx`, jsdom): 카드 렌더, 절정/제철 UI 부재,
  빈 상태 카피. `NavIndex`에 축산물 링크가 뜨고 라우팅되는지(`renderWithRouter`).
- **타입 픽스처 유효성**: `categoryCode: '500'`, `category: 'livestock'`는 유니온에 추가된
  값이어야 `tsc --noEmit` 통과.
- **완료 게이트**: `npm test` **와** `npx tsc --noEmit` 둘 다. UI는 `npm run dev`로 브라우저 실측
  (탭 진입·카드·계절 UI 부재·빈 상태). 사용자향 시각 변경이면 스크린샷 사인오프.

## 범위 밖 (YAGNI)

- 축산물의 "다가오는" 뷰 — 제철이 없어 불필요.
- 등급을 한 카드에 여러 줄로 접는 새 카드 모양 — 등급마다 별도 카드로 결정됨.
- 지역·도매·산지 가격 — 소매 전국 기준 유지(기존 정책과 동일).
- 개인화·알림·즐겨찾기 — 다음 사이클.
