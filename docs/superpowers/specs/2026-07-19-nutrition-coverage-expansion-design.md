# 영양 정보 커버리지 확장 — 설계

*작성 2026-07-19. 3개 확장 작업(수산물·축산물·영양) 중 첫 번째 — "쉬운 것부터" 순서.
관련: [아이디어 백로그](../../아이디어-백로그.md), [제품 동작 지도 §8](../../제품-동작-지도.md).*

## 문제

영양 정보는 40개 프로필 중 **3종(복숭아·토마토·사과)에만** 있다. 나머지 37종은 카드를
펼쳐도 영양 스탯이 없고, 영양 있는 품목이 이번 달 제철이 아니면 푸터의 출처 줄까지 사라진다
(`제품-동작-지도.md §8`). 나머지 품목의 영양 참조를 채워 커버리지를 넓힌다.

## 제약: 나는 식약처 키가 없다

영양 매칭은 `foodDb.foodName`이 식약처 영양성분DB의 `FOOD_NM_KR`과 **정확 일치**해야 한다
(`src/types.ts` `FoodDbRef`, `scripts/lib/parse-nutrition.mjs`가 `x.FOOD_NM_KR === foodName`
로 집는다). 그런데 이 작업을 하는 에이전트는 `DATA_GO_KR_KEY`(CI 시크릿)가 없어 DB에 실제로
어떤 `FOOD_NM_KR` 문자열·수치가 있는지 확인할 수 없다. 문자열을 추측하면 틀리는 순간
**조용히 매칭 실패 → 카드에서 영양 줄이 사라진다**(부재로 나타나는 놀라움). 그래서 정확한
문자열을 **DB에서 실측**한 뒤 참조를 집는 단계가 설계에 반드시 들어간다.

## 데이터 흐름 (3단계 · 유저 API 실행 2회)

```
[1] 탐색   npm run explore:nutrition   (유저가 실행, DATA_GO_KR_KEY 필요)
             → 37종을 식약처 API로 조회, 변형 후보 + 6대 수치 + 실제 category1 수집
             → 선택정책으로 순위 매겨 nutrition-candidates.json 출력 (gitignore)
[2] 참조 집기 (에이전트)
             → 후보 파일을 읽고 품목별 foodName·category1 확정 → produce.json의 foodDb 채움
             → 대표 없는 품목은 foodDb 비운 채 유지(정직한 부재), 이 문서·커밋에 명시
[3] 수집   npm run fetch:nutrition      (유저가 실행, 기존 스크립트 그대로)
             → 이제 채워진 참조를 다 찾아 nutrition.json 생성 → 커밋
```

유저 API 실행이 두 번(탐색 → 수집)이다. 영양은 씨앗형(상시 CI 없이 로컬 1회)이라 API를 두 번
때려도 무해하지만, 핸드오프가 두 번이라는 걸 감안한다.

## 선택 정책 (카드 수치를 좌우 — 데이터 정책으로 표면화)

식약처 DB엔 한 품목에 여러 변형이 있다(품종·조리상태: `사과_부사_생것` vs `사과_구운것`).
어느 변형을 고르냐가 카드에 뜨는 수치를 정한다. 정책:

- **생것(raw) 우선.**
- 생것이 없으면 **데친것·삶은것 등 최소조리 허용**(커버리지 우선). 단 조리 기준을 쓴 품목은
  탐색 출력에 **플래그**로 표시해, "이건 삶은 기준"임을 의식적으로 알고 넘어간다.
- **중가공 제외** — 통조림·주스·말린것·냉동·설탕가공 등.
- 품종은 **가장 대표적인 것 하나**. 없으면 가장 평범한 이름.
- 어떤 변형도 적절치 않으면 **그 품목은 건너뛴다**(`foodDb` 비움 = 오늘과 같은 정직한 부재).
  탐색 출력에 `no-match`로 명시해 조용히 빠지지 않게 한다.

트레이드오프: 삶은 기준 품목은 카드 간 "생것 vs 조리" 기준이 섞인다. 커버리지를 택한
결과이며, 어떤 품목이 조리 기준인지는 이 문서와 탐색 출력이 기록으로 남긴다.

## category1 필터 (조용한 누락 방지)

`FoodDbRef.category1`은 API의 `FOOD_CAT1_NM` 필터값이다. 현재 3종은 `과일류`/`채소류`만
쓴다. 하지만 감자·고구마는 `감자류`/`서류`, 옥수수는 `곡류` 등 **채소류가 아닌 분류**일 수
있다. `채소류`로 하드코딩하면 이들이 조용히 매칭 실패한다. 그래서:

- 탐색 스크립트는 category1로 과하게 좁히지 **않고**, 응답의 **실제 `FOOD_CAT1_NM`을 캡처**한다.
- 참조에는 그 실제 값을 담는다. `FoodDbRef.category1`의 주석/타입이 `과일류`/`채소류` 두 값만
  가정한다면, 이 작업에서 실제 등장하는 값들을 반영해 넓힌다(구현 시 확인).

## 새 컴포넌트

- **`scripts/explore-nutrition.mjs`** (커밋형, `package.json`에 `explore:nutrition`) —
  개발용 탐색 도구. 나중에 수산물 영양 확장에도 재사용. `DATA_GO_KR_KEY` 필요.
  - 입력: `produce.json` 중 `foodDb` 없는 프로필 + 품목별 검색어 매핑(스크립트 내 상수).
  - 각 품목을 `FOOD_NM_KR`로 조회해 변형 후보 전부 + 6대 수치 + `FOOD_CAT1_NM` 수집.
  - **부분일치 폴백**: API가 `FOOD_NM_KR` 부분일치를 지원하면 그대로, 정확일치만 지원하면
    카테고리로 넉넉히 받아(`numOfRows` 크게) 클라이언트에서 이름 포함으로 필터.
  - 선택정책으로 순위 매겨 품목별 **추천 픽 + 전체 후보 + 플래그(cooked/no-match)**를
    `nutrition-candidates.json`(gitignore)로 출력.
- **`scripts/lib/rank-nutrition-candidates.mjs`** — 순위·플래그 판정 **순수 함수**. I/O 셸과
  분리(기존 관례). `tests/`에 Vitest 테스트(생것 우선, 조리 폴백, 중가공 제외, no-match).

## 건드리지 않는 것

- **`fetch-nutrition.mjs`·`parse-nutrition.mjs`** — 로직 변경 없음. `parseNutritionEntry`가
  이미 `FOOD_NM_KR` 정확일치로 집는다. 유일한 실질 변경은 **데이터**(참조의 정확한 문자열·
  실제 category1)다.
- **`schemaVersion`** — `NutritionSnapshot`은 `1` 그대로. 새 필드 없음(kcal·탄수·단백·지방·
  당류·식이섬유 6개 유지). 스키마 가드가 필요한 구조 변경이 아니다.
- **UI 컴포넌트** — 변경 없음. 기존 렌더링 경로에 데이터만 더 흐른다. `hasNutrition`(app.ts)이
  이번 달 제철 품목 중 영양 있는 게 있으면 출처 줄을 켠다 — 커버리지가 늘면 더 자주 뜬다.

## 검증 (완료 게이트)

- `npm test` **와** `npx tsc --noEmit` 둘 다 통과.
- 순위 순수함수 테스트가 정책(생것 우선·조리 폴백·중가공 제외·no-match)을 덮는다.
- 수집 후 `nutrition.json` 엔트리 수 확인(3 → N).
- **브라우저 실측**: `npm run dev`로 7월 제철 중 새로 커버된 카드(오이·감자·양파·애호박 등)를
  펼쳐 영양 스탯 + 푸터 출처 줄이 렌더되는지 확인. 사용자향 시각 변경이므로 스크린샷 사인오프.
