# Storybook — 뷰 상태 탐색기 설계

날짜: 2026-07-14

## 왜

이 앱의 화면은 **조건부 슬롯의 모음**이다. 슬롯마다 자기를 켜는 데이터 조건이 따로 있고,
어느 날의 데이터는 그중 일부만 켠다. 꺼진 슬롯은 코드에 살아 있지만 화면에 없다 —
그래서 데이터가 바뀌는 날 "예상치 못한 UI 요소"로 튀어나온다.

2026-07-14 실측(`buildAppView`를 진짜 데이터로 직접 호출):

- 7월 제철 프로필은 19개인데 **카드는 5개만** 렌더된다 (`selectPicks`의 `limit = 5`).
  "7월의 제철" 칩 목록에는 19개가 다 뜨므로, 칩 수와 카드 수가 다른 게 정상이다.
- 오늘 켜진 슬롯: 절정 점 · 가격 블록 · 취소선 예전가 · 하락 칩(쪽빛) · 스파크라인 · 레시피 칩.
- 오늘 **꺼진** 슬롯: 상승 칩(러스트) · "한 달 전과 비슷해요" · 개당값 줄 · 영양 스탯 6칸 ·
  stale 경고 · noDrop 안내 · 빈 상태 · 가격 없는 카드.
- 가장 은밀한 것: **영양 정보는 프로필 40개 중 3개(복숭아·토마토·사과)에만 있고**, 그 셋 중
  아무도 오늘의 top-5에 못 들었다. 그래서 카드의 영양 스탯뿐 아니라 **푸터의 "영양: 식약처…"
  출처 줄까지 페이지에서 통째로 사라진다** (`app.ts`의 `hasNutrition`이 카드 기준이므로).
  복숭아 가격이 조금 더 떨어져 top-5에 드는 날, 처음 보는 스탯 그리드와 푸터 줄이 함께 생긴다.

이걸 문서로 적어두는 것으로는 부족하다. 읽어도 화면이 안 그려진다.
**손으로 돌려보며 인과를 확인할 수 있는 도구**가 필요하다.

## 무엇을

Storybook 10 + `@storybook/react-vite`. 진짜 컴포넌트·진짜 CSS 위에서,
**원시 데이터 노브를 돌리면 진짜 파생 파이프라인이 작동해** 카드가 변한다.

## 핵심 결정 — 노브는 `CardView`가 아니라 원시 재료다

`ProduceCard`가 받는 `CardView`는 `card.ts`가 **계산을 끝낸 결과물**이다.
여기에 controls를 붙이면 결과를 손으로 조립하는 셈이라, 정작 보고 싶은 인과
(`변화율 −0.4% → 칩이 사라지고 "비슷해요"가 뜬다`)가 안 보인다.

그래서 스토리는 재료를 짓고 **진짜 파이프라인에 통과시킨다**:

```
노브(현재가 · 한 달 전 · 작년 · 단위 · 절정 · 영양 유무 · 레시피 개수)
  └→ ProduceProfile + PriceEntry            (재료)
       └→ priceView() → PickResult          [picks.ts — 진짜]
            └→ toCardView()                 [card.ts — 진짜]
                 └→ <ProduceCard>           [진짜 컴포넌트 · 진짜 CSS]
```

한 달 전 대비를 −0.4%로 돌리면 스토리가 아니라 `card.ts`의 `Math.abs(pct) < 1`이 칩을 지운다.
보이는 것은 목업이 아니라 **실제 규칙**이다.

부수 효과 하나가 특히 값지다: 도달 불가능한 조합(예: 무게 단위인데 개당값이 있는 카드)은
**애초에 만들 수 없다.** 파이프라인이 그것을 허락하지 않기 때문이다.

### `vite.config.ts`는 상속하되 `tanstackStart`만 벗긴다

`tanstackStart()`는 SSR·프리렌더·라우트 코드젠을 하는 플러그인이라 Storybook의 클라이언트 전용
개발서버와 싸운다. 별도 `vite.config`를 복제하지 않고, `viteFinal`에서 이 플러그인만 필터한다 —
복제본은 본 config가 바뀔 때 조용히 밀리고, 이 프로젝트는 그런 어긋남을 싫어한다.

`tanstackStart()`는 플러그인 하나가 아니라 **26개짜리 프리셋**이다(실측). 이름이 전부
`tanstack-` 접두어를 쓰므로 접두어로 거른다. `plugins` 배열은 중첩될 수 있어 먼저 평탄화한다.

**하나도 못 걸러내면 throw한다.** 플러그인 이름 규칙이 바뀌어 필터가 헛돌면 Storybook이
조용히 이상하게 동작하는 대신 시끄럽게 실패해야 한다 (`CONTEXT.md`의 KAMIS 어댑터 원칙과 같다:
*조용한 오염보다 시끄러운 실패가 낫다*). 개수를 26으로 못 박지는 않는다 — 버전마다 달라질 값이고,
그걸 규칙으로 굳히면 업그레이드마다 헛되이 깨진다.

### 두 가지 렌더 제약

- **`#app` 래퍼가 필수** — `style.css`의 `#app { max-width: 28rem }`이 모바일 폭을 만든다.
  없으면 스토리가 데스크톱 전폭으로 퍼져 실물과 다르게 보인다.
- **`<Link>`를 쓰는 스토리는 라우터가 필요** — `NavIndex`가 TanStack `<Link>`를 쓰므로
  `App`·`Coming` 스토리는 메모리 라우터로 감싸지 않으면 크래시한다.
  `test-utils.tsx`의 메모리 라우터 패턴을 재사용한다. 카드 스토리는 라우터가 필요 없으므로
  데코레이터를 전역이 아니라 **스토리별로** 붙인다.

### 계절 축

`__root.tsx`가 `<body data-season={seasonOf(현재월)}>`를 달고, `style.css`가
`[data-season]`에 `--accent`/`--tint`를 매단다 (봄 연두 · 여름 노랑 · 가을 오렌지 · 겨울 자주).
같은 카드도 12월엔 다른 색이다. 전역 데코레이터가 실제 앱과 똑같이 `document.body`에
`data-season`을 세팅하고, Storybook 툴바 스위치로 네 계절을 갈아끼운다.

## 월 노브의 정직한 한계

`prices.json`은 **한 장의 스냅샷**(2026-07-13 조사분)뿐이다. 12월로 돌려도 가격 숫자는 7월 값이고,
그대로 두면 `staleDays`가 150일이 되어 모든 월에 stale 경고가 붙는다.

그래서 이 스토리는 **`surveyedOn`만 시뮬레이션 날짜로 덮어쓴다.** 제철 선정·정렬·카드 구성·
계절 팔레트는 전부 진짜고, **가격 숫자만 "7월 실측 참고용"**이다. 이 한계를 스토리 설명에 적는다.
가격 신선도는 이 스토리가 아니라 **stale 전용 스토리**에서 정면으로 본다.

## 게이트

- `npx tsc --noEmit` — 스토리가 `src/` 아래라 `tsconfig.include`에 자동 포함된다.
  뷰 타입이 바뀌면 **스토리가 타입에러로 먼저 깨진다** — 스토리가 문서가 아니라 계약이 된다.
- `npm test` — 영향 없음. Vitest 기본 include는 `*.test.*`만 잡으므로 `*.stories.tsx`는 안 섞인다.
- `npm run storybook`을 띄워 **브라우저로 실측**한 뒤 완료로 부른다 (CLAUDE.md 검증 게이트).

**CI에는 넣지 않는다.** 이건 배포물이 아니라 그림을 되찾으려고 만드는 도구다.
`tsc`가 계약 위반을 잡아주는 것으로 충분하다.

## 하지 않는 것

- `PriceBlock`·`NutritionLine` 등 하위 컴포넌트의 개별 스토리 — `ProduceCard` 하나가 전부 덮는다.
- `play` 함수 기반 상호작용 자동 재생(레시피 메모 열기·절기 dot 툴팁·NavIndex 차양) —
  Storybook에서 손으로 직접 만져볼 수 있다. 필요해지면 그때.
- 비주얼 회귀 스냅샷 — 지금 필요한 건 회귀 감시가 아니라 이해다.
