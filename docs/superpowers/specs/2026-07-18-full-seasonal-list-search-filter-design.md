# 제철 전체 목록 · 검색 · 필터 — 설계

*근거: [제품 동작 지도](../../제품-동작-지도.md)의 문 3(top-5)·수집 정책, [DESIGN.md](../../DESIGN.md)의
냉장고-메모 카드 결정(2026-07-11), [CLAUDE.md](../../../CLAUDE.md)의 데이터 정책 규칙.*

작성: 2026-07-18. 브레인스토밍에서 사용자가 네 갈래 불만 중 **"왜 5개만 보이나"**를 골라 좁혔다.

## 왜 (문제)

`/`(지금 담기 좋은 것)는 세 문을 통과한 뒤 **상위 5장**만 카드로 만든다(`selectPicks`의 `limit=5`).
정렬이 **절정 먼저 → 하락 큰 순**이라, 절정이 많은 달(6·7월, 절정 10~13종)엔 **절정이 5칸을 독식**한다.
그래서 절정 아닌데 크게 싸진 품목(예: 7월 방울토마토 −26%, 참외 −20%)이 카드가 못 되고 하단
"이번 달 제철" 이름 칩으로만 밀린다. 사용자 관찰:

- **5라는 수가 자의적이다** — 사용자가 정하지 않았고, "왜 5개?"의 답이 화면에 없다.
- **맞는 게 숨는다(문제의 핵심)** — 싸진 품목이 절정에 가려 안 보인다. "분명 싸졌는데 왜 없지"의 정체.
- **취향이 강한 사람은 자기 품목부터 확인하고 싶다** — 지금은 훑거나 찾을 길이 없다.

방향: **top-5 컷을 걷고 이번 달 제철 전체를 보여주며, 검색·필터·정렬로 좁히게 한다.** 자의적 top-5야말로
앱이 대신 편집하는 "거울스러운" 움직임이고, 전체를 펼쳐 필터로 좁히는 게 오히려 "이번 달 계절 전체를
비추는 창"에 가깝다 — [북극성](../../../CLAUDE.md)(창이지 거울)과 어긋나지 않는다(사용자 확인).

## 화면 동작

### 정렬 — 하락 큰 순 (기본)

각 픽을 그룹으로 나눈 뒤 그룹 순, 그룹 안에서 등락률 오름차순(가장 음수가 위):

```
group(r) = 가격 없음        → 2   (맨 뒤)
           가격 있고 등락 모름 → 1   (기준선 없음: 예 복숭아 — 한 달 전 값 결측)
           가격·등락 있음     → 0
정렬: group 오름차순; group 0 안에서는 changeVsMonthAgoPct 오름차순
      (−26% → −11% → +2% → +13% : 하락 먼저, 상승은 자연히 아래로)
```

- **절정(`inPeak`)은 더 이상 정렬에 관여하지 않는다.** "절정이 5칸 독식" 놀라움이 사라진다.
- **비슷(±1%)** 은 등락률이 0에 가까우니 group 0 안에서 상승·하락 사이에 자연스럽게 앉는다.
- 다른 정렬(이름/가격)은 사용자가 컨트롤로 선택. 이름=`localeCompare(…, 'ko')`; 가격 낮은 순=
  `price` 오름차순, 무가격 맨 뒤.

### 필터 (이번 달 제철 목록에만 작동)

순수 술어의 AND. 과일/채소만 상호배타(둘 다 켜면 마지막 것):

| 필터 | 술어 |
|---|---|
| 과일만 | `category === 'fruit'` |
| 채소만 | `category === 'vegetable'` |
| 내려간 것만 | `price != null && change != null && change < 0` |
| 절정만 | `inPeak` |
| 가격 있는 것만 | `price != null` |

### 검색 (프로필 40종 전체)

- **빈 검색** → 평소 목록(필터·정렬 적용).
- **검색어 있음** → 이름 부분일치(`name.includes(q)`)로 40종을 뒤져 둘로 가른다:
  1. **이번 달 제철 매치** → 카드로(필터·정렬은 이 부분에만 적용).
  2. 그 아래 **은은한 구분선 "지금은 제철이 아니에요"**, 그리고 **비제철 매치는 가벼운 힌트 행**:
     `🍓 딸기 · 12~4월 제철` (가격·펼침 없음). 그 품목의 다음 제철이 `/coming` 범위(2개월) 안이면
     "다가오는 제철에서 보기" 링크(`/coming`)를 얹는다.
- 비제철 힌트 행에는 **필터를 적용하지 않는다**(이름 매치만) — 다른 축(전체 달력)의 안내라서.
- `seasonLabel(seasonMonths)` 순수 함수로 `[12,1,2,3,4] → "12~4월"` 같은 연속 구간 라벨을 만든다
  (여러 구간이면 `,`로 잇는다).

### 빈 화면 (DESIGN.md 목소리 — 담백, 느낌표 없음)

- 검색 무매치(제철·비제철 모두 없음): `'○○' 제철 품목을 찾지 못했어요`.
- 필터 결과 없음(검색 없음): `조건에 맞는 제철 품목이 없어요`.

## 카드 변경 (시안 B — 최소)

**냉장고-메모 카드를 그대로 둔다.** 배경·테두리·마스킹테이프·±0.4° 기울기·`--lift` 부양·손글씨
`.why`·`PriceBlock`·`<details>` 펼침(손질법·영양·레시피·스파크) **전부 유지**. 유일한 변경:

- **`.card::before`(테이프) 축소** — 폭 `4.6rem → ~3.4rem`, 높이 `1.3rem → ~1rem`(정확값은 브라우저
  실측). 카드가 19장으로 늘어나면 큰 테이프가 반복돼 시끄러우므로 살짝 줄인다.

세로 밀도는 **현행 유지**(패딩 안 줄임) — 사용자가 "변경 최소"를 이유로 B를 골랐으므로 조이지 않는다.

## 아키텍처 경계별 변경

경계는 CLAUDE.md의 지도를 따른다(순수는 `picks`/`card`/`app`, 표시는 `components`).

### `src/picks.ts` (순수 — "무엇을 고르나")

- **`selectPicks`**: `limit` 제거(전체 반환). 정렬을 **하락 큰 순**(위 규칙)으로. `inPeak`는 필드로 유지.
  (정렬 우선순위가 바뀌므로 함수 주석·제품-동작-지도의 지렛대 지도도 갱신.)
- **신규 순수 헬퍼**:
  - `filterPicks(picks, filters): PickResult[]` — 위 술어의 AND.
  - `sortPicks(picks, mode): PickResult[]` — `'drop' | 'name' | 'priceLow'`.
  - `searchProduce(profiles, month, query): { inSeason: ProduceProfile[]; offSeason: OffSeasonHint[] }`
    — 40종 이름 매치를 제철/비제철로 가른다.
  - `seasonLabel(months: number[]): string` — 연속 구간 → `"12~4월"`.

### `src/app.ts` (순수 — 뷰 조립)

- `buildAppView`가 **전체 제철 카드**를 싣는다(정렬은 기본 하락 큰 순으로 미리 적용 — 무JS 폴백이 이
  순서로 렌더된다).
- `AppView`에 **검색 인덱스** 추가: 비제철 프로필의 경량 뷰(`OffSeasonHint[]`: `emoji`, `name`,
  `seasonLabel`, `comingSoon`). 제철 매치는 이미 `cards`에 있으니 클라이언트가 `cards`를 직접 거른다.
  `comingSoon`은 `comingMonths(profiles, month)` 범위 포함 여부.
- `noDrop`·`hasNutrition`·`hasRecipes`는 이제 **전체 카드 기준**으로 계산(값이 더 정확해진다 — 영양
  3종·레시피 20종이 제철이면 대개 출처 줄이 뜬다).
- 하단 이름 칩용 `seasonal`은 **제거**(칩 목록을 없애므로).

### `src/view-types.ts`

- `AppView`에 `searchIndex: OffSeasonHint[]` 추가, `seasonal` 제거. `OffSeasonHint` 타입 신설.

### `src/components/`

- **`App.tsx`** — 검색·필터·정렬을 **클라이언트 상태**(`useState`)로 쥔다. 상단에 `SearchBar` +
  `FilterBar` + `SortControl`, 그 아래 걸러진 카드 목록, 검색 시 비제철 힌트 섹션. 하단 칩 목록 삭제.
- **`ProduceCard.tsx`** — 구조 변경 없음. (테이프 축소는 CSS만.)
- **신규 소형 컴포넌트**: `SearchBar`, `FilterBar`(칩 토글), `SortControl`(셀렉트/세그먼트),
  `SeasonHint`(비제철 힌트 행).
- **과일/채소 필터**는 기존 라디오+CSS(`#f-fruit`/`#f-veg`)에서 **JS 상태로 이전** — 검색·정렬이
  이미 JS라 필터만 CSS로 두면 두 진실이 갈린다. `FilterBar`가 한 곳에서 관장.

### `src/style.css`

- `.card::before` 축소. 컨트롤(`SearchBar`/`FilterBar`/`SortControl`) 스타일 — 쪽빛 규율·칩 어휘 재사용
  (검색창 `--card` 배경·`--line` 테두리·포커스 쪽빛 2px; 필터 칩은 기존 `.chip` 계열, 켜짐은 `--tint`).
- 무JS일 때 컨트롤을 숨기거나(`.js-only`) 비활성 표기 — 목록 자체는 항상 보인다.

### 점진 향상 / 무JS

- 프리렌더가 **전체 목록을 기본정렬(하락 큰 순)로 이미 그린다.** JS 없이도 전체가 보인다(오늘의 5장보다
  나음). 검색·필터·정렬은 하이드레이션 후 붙는 향상.
- 공개 페이지 규칙(런타임 외부요청 없음·무추적)은 유지 — 모든 상호작용은 클라이언트 로컬, 네트워크 없음.

## 결정 기록

- **시안 C → B 번복.** 처음 리스트 행(C, 밀도 최고)을 골랐으나, before/after 검토에서 냉장고-메모
  크롬(테이프·기울기·부양) 제거가 **변경 폭이 크다**고 판단해 B(현행 메모 유지)로 되돌림. 정체성을
  지키고 변경을 최소화하는 선택. 밀도는 카드 수를 늘리는 것만으로 얻고, 카드 자체는 오늘 그대로.
- **메모-카드 결정(2026-07-11)은 유지.** 이번 변경은 카드 룩이 아니라 **목록 정책**(cap·정렬·컨트롤)이다.
  테이프 축소만 그 결정의 미세 조정.
- **절정이 정렬 키에서 빠짐.** "지금 담기 좋은 것"의 순서를 "절정 먼저"가 아니라 "싸진 것 먼저"로 바꾼다
  — 사용자 통증(싸진 게 숨음)의 직접 해소. 절정은 정보(●)·필터로 남는다.

  미리 수집·커밋 + 클라 선택" 또는 개인화가 필요 — 다음 사이클. 이번엔 "· 전국 평균" 라벨로 표면화만.
