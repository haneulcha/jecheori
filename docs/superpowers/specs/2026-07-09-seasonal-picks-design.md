# wat-to-buy — 이번 주 제철 픽 디자인

날짜: 2026-07-09
상태: 사용자 승인됨 (구현 계획 작성 전)

## 한 문장 정의

장 보러 가기 전(또는 장 보는 중에) 열어서, "지금 안 먹으면 아쉬운 것" 몇 가지를
30초 안에 확인하는 모바일 웹 페이지.

이커머스가 아니라 계절을 알려주는 달력 같은 담백한 톤. 광고·로그인·추적 없음.

## 배경과 데이터 소스 결정

처음 검토한 KADX "알뜰 장보기 물가정보" 상품
(https://kadx.co.kr/opmk/frn/pmumkproductDetail/PMU_c376889c-39b7-48de-92cd-e3dcaa608c67/6)은
OPEN API가 아니라 **월간 갱신 ZIP 다운로드**(주간 인포그래픽 PDF 묶음, 2.0GB)로
확인되어 제외했다. 대신 같은 aT가 운영하는 **KAMIS 오픈API**(일별 품목별
소매가격, 무료, 매일 갱신)를 데이터 소스로 사용한다.

- KAMIS 오픈API: https://www.kamis.or.kr (키 발급 필요)
- 사용 엔드포인트: 일별 품목별 소매가격 (전국 평균 기준)
- 가격 신호 해석: "1개월 전 대비 하락 = 물량이 쏟아지는 시기 = 지금이 때"
  (aT의 "알뜰한끼" 서비스와 같은 접근)

## 범위

- 품목: 과일 + 채소 (~40개 품목으로 시작). 수산물·축산물은 범위 밖.
- 형태: 모바일 우선 웹앱 (정적 사이트). 네이티브 앱·푸시·다이제스트는 범위 밖.
- 큐레이션 콘텐츠: 직접 작성한 정적 제철 프로필. LLM 생성은 사용하지 않음.

## 사용자 경험

단일 페이지, 모바일 우선.

1. **상단 헤더**: 시기 인사말 (예: "7월 둘째 주, 지금 장바구니에 담기 좋은 것들")
2. **제철 픽 카드 3~5장**, 각 카드는:
   - 품목명 + 제철 배지 (예: "🍑 복숭아 · 제철 한창")
   - 왜 지금인지 한 줄 (예: "7~8월이 노지 복숭아의 절정이에요")
   - 가격 신호: 오늘 전국 평균 소매가 + 1개월 전 대비 등락
     (예: "4,800원/kg · 한 달 전보다 12% ↓")
   - 탭하면 펼쳐짐: 고르는 법 / 보관법 / 간단한 활용법
3. **하단**: "이번 달 제철 전체 보기" — 이번 달 제철 품목 전체의 심플한 리스트

## 제철 픽 선정 로직

프론트엔드에서 두 JSON(제철 프로필 + 가격 스냅샷)을 합쳐 순수 함수로 계산한다.

1. 제철 프로필에서 **이번 달이 제철인 품목** 필터 (절정 월이면 가중치)
2. **1개월 전 대비 하락률**이 큰 순으로 정렬
3. 상위 5개를 픽으로 선정. 가격 데이터가 없는 품목은 제철 정보만으로 뒤에 배치

## 아키텍처: 정적 사이트 + 스케줄 데이터 갱신

서버 없음. GitHub Actions가 매일 KAMIS를 호출해 JSON 스냅샷을 굽고,
GitHub Pages로 정적 배포한다. KAMIS API 키는 CI 시크릿에만 존재한다.

검토한 대안: (B) 서버리스 프록시 — 동적 조회는 유연하지만 이 규모에 과함,
(C) 풀스택 + DB — 유지비가 "소박한 앱" 방향과 상충. 모두 기각.

### 저장소 구조

```
wat-to-buy/
├── data/
│   ├── produce.json        # 직접 작성: 품목별 제철 프로필 (~40개 품목)
│   └── prices.json         # 파이프라인 생성: KAMIS 가격 스냅샷
├── scripts/
│   └── fetch-prices.mjs    # KAMIS 호출 → prices.json 생성 (Node, 의존성 최소)
├── src/                    # Vite + vanilla TypeScript (프레임워크 없음)
│   ├── main.ts             # 렌더링
│   ├── picks.ts            # 선정 로직 (순수 함수)
│   └── style.css
└── .github/workflows/
    ├── update-prices.yml   # 매일 07:00 KST cron → 스크립트 → 변경 시 커밋
    └── deploy.yml          # GitHub Pages 배포
```

### 데이터 스키마

`data/produce.json` — 품목당:

| 필드 | 설명 |
|---|---|
| `id`, `name`, `emoji` | 식별자, 표시명, 이모지 |
| `kamisCode` | KAMIS 품목 코드 (부류/품목/품종) |
| `seasonMonths` | 제철 월 배열 (예: `[6,7,8]`) |
| `peakMonths` | 절정 월 배열 (예: `[7,8]`) |
| `whyNow` | 월별 한 줄 문구 (월 → 문자열 맵, 없으면 기본 문구) |
| `howToPick` | 고르는 법 |
| `howToStore` | 보관법 |
| `howToUse` | 간단한 활용법 |

초안은 Claude가 작성하고 하늘이 다듬는다.

`data/prices.json` — 품목당:

| 필드 | 설명 |
|---|---|
| `price` | 오늘 전국 평균 소매가 |
| `unit` | 단위 (예: kg, 개) |
| `priceMonthAgo` | 1개월 전 가격 |
| `priceYearAgo` | 1년 전 가격 |

파일 상단에 `fetchedAt` (조회 일시) 포함.

## 에러 처리

- **파이프라인**: KAMIS 호출 실패 시 `prices.json`을 건드리지 않고 워크플로만
  실패로 표시한다. 앱은 마지막 성공 스냅샷으로 계속 동작한다.
- **오래된 데이터**: `fetchedAt`이 3일 이상 지났으면 프론트에 "가격은 N일 전 기준"
  배지를 표시한다.
- **품목별 결측**: 특정 품목이 "조사 없음"이면 그 품목만 건너뛰고, 카드에서는
  가격 줄만 생략하고 제철 정보는 그대로 보여준다.

## 테스트 (Vitest)

- 선정 로직 순수 함수 유닛 테스트: 제철 필터, 절정 가중치, 하락률 정렬,
  가격 결측 처리
- KAMIS 응답 파싱: 실제 응답을 픽스처로 저장해 API 콜 없이 테스트

## 사전 준비물

- KAMIS 오픈API 키 발급 (www.kamis.or.kr에서 신청, 보통 즉시~수일).
  키가 나오기 전에는 픽스처 데이터로 개발한다.
- GitHub 저장소 + Pages 활성화, 키를 Actions 시크릿(`KAMIS_API_KEY` 등)으로 등록.
